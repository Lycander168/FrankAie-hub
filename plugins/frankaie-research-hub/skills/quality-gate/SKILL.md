# 產出品質閘門 / 對抗式審查（Quality Gate）

每一份要交出去的產出，先過這道閘門。原則一句話：

> **早審、常審（review early, review often）；通過驗證才算完成（verified, not assumed）。**
> 在問題擴散到下游、累積成大返工之前，先用懷疑的眼光把它擋下來。

> 本 skill 改寫自開源社群 superpowers 的 `requesting/receiving-code-review` 與
> `verification-before-completion` 方法（`obra/superpowers`），並在地化為 HUB 的「產出（deliverable）審查」。

---

## 何時啟動（強制觸發點）

- **每階段產出後**：三階段 SOP 任一階段（搜集 / 驗證 / 開發）產出文件後，先審再進下一階段。
- **交棒 / 交接前**：角色把產出交給下一棒角色，或交給營運/財務/行銷中心之前。
- **結案前**：`hub-router` 宣告專案完成前，必過閘門。
- **使用者點名**：「審一下 / 驗收 / 挑毛病 / 這份可以了嗎」。

> 紅線：**不要因為「看起來很簡單」就跳過審查**；不要在有「致命」未解時繼續；不要無理由駁回有技術依據的意見。

---

## 嚴重度三級（Severity）

| 級別 | 定義 | 處置 |
|------|------|------|
| 🔴 **致命 (Critical)** | 結論會錯、會誤導決策、有安規/法規/臆造風險 | **立刻修**，未修不得放行 |
| 🟡 **重要 (Important)** | 不致命但會降低品質/可信度，下游會踩雷 | **進下一步前修完** |
| 🟢 **次要 (Minor)** | 潤飾、格式、可日後優化 | **記錄**進 `progress.md`，擇期處理 |

## 審查維度（HUB 專用檢核表）

對著產出逐項打勾，任一不過就開 issue：

1. **防臆造 / 來源**：每筆關鍵主張是否有來源 + 日期？無一手資料是否確實標 `【需查證】`？有沒有把推測寫成事實？（🔴）
2. **可回溯 / 可驗證**：數據、效率、溫度、認證、銷量等是否能回溯到來源或測試方法？（🔴）
3. **可量測**：規格/功能是否轉成可量測指標（W/V/A/℃/mAh/效率%/尺寸/轉換率）？而非模糊形容詞？（🟡）
4. **邏輯一致**：結論與前面的數據/風險是否自洽？有沒有自相矛盾或跳步？（🔴）
5. **風險揭露**：關鍵風險（發熱、EMI、認證卡關、專利、法規、退貨）是否點出且附緩解？（🟡）
6. **範本合規**：是否套用該角色對應輸出範本、欄位齊全（情報卡/驗證報告/Dev Package…）？（🟢）
7. **驗證充分（限消費者測試）**：購買意願評分是否有樣本與市場標註，而非單一拍腦袋？（🟡）
8. **可交接**：下一棒/下一個中心拿到這份，能不能直接接著做、不用回頭重問？（🟡）

---

## 完成前驗證（Verification Before Completion）

宣告「完成」前，必須能對下面每一條回答「是」：

- [ ] 所有審查維度的 🔴 致命項都已修正且複查過？
- [ ] 🟡 重要項已處理或已明確記錄為「已知限制」？
- [ ] 產出有套用對應範本、欄位無缺漏？
- [ ] 關鍵主張都有來源 / 測試方法 / `【需查證】` 標註？
- [ ] `task_plan.md` 對應階段可以從 `in_progress` 改成 `done`？

> 只要還有任一條為「否」，就**不是完成**——回去補，不要提早收尾（呼應 `planning-with-files` 的完成閘門）。

---

## 收到審查意見時的回應法（給被審的角色）

對抗式審查不是吵架，是把產出變更好。被審角色依序：

1. **讀完**：先完整吸收意見，不要急著反駁。
2. **理解**：用自己的話複述要求；不清楚就先問，不要半懂就改。
3. **查證**：拿意見去對照實際數據/來源，確認是否成立。
4. **評估**：在「這個案子的脈絡」下判斷技術是否站得住。
5. **回應**：用事實或技術理由回應——**不說「你說得對！」這種場面話**；產出本身證明你懂了。
6. **實作**：一次處理一項，改完重新驗證。

> 心法：**技術正確優先於社交舒適。** 錯了就事實性地更正，不過度道歉；要反駁就拿數據/測試/來源，不要靠嘴硬。
> 對「外部建議」保持健康懷疑：先確認不會弄壞既有結論、脈絡有被完整理解、且這個功能/主張真的會用到（YAGNI）。

## 與 LYCANDER GROUP 的協作介面

- **與 `planning-with-files`**：審查結論寫進 `progress.md`，未解問題回寫 `findings.md` 的「待查證」，
  階段狀態唯有通過閘門才在 `task_plan.md` 標 `done`。
- **與 `hub-router`**：路由在每個階段交棒前插入一次 quality-gate；結案前做最後總驗收。
- **與各角色**：審查只挑錯、要證據、擋未完成，**不替角色重寫專業內容**；修正由原角色執行。

## 啟動方式

收到一份待審產出時：對照「審查維度」逐項檢查，標出 🔴🟡🟢，產出《審查報告》，
再跑「完成前驗證」檢核表決定是否放行。不放行就把球踢回原角色並指出具體修法。

> 提醒：閘門本身也守防臆造原則——指出問題時要附「證據/出處」，不可憑感覺說「這裡怪怪的」。
